记录一次排查因线程WAITING导致应用假死的经历

发表于2024-10-16|工作log
|字数总计:1.9k|阅读时长:9分钟|阅读量:|评论数:

背景

在一个阳光明媚的下午，我正在悠哉游哉的在工位上畅游网络世界(摸鱼 😃 )，忽然心里有点不好的预感，果然哈，企业微信里产品经理弹了一条消息给我 “测试服务器的系统登录进去获取不到菜单，急速度处理”，我小嘴一撇快速回复“好，小问题”，然后连接到测试服务器，到系统管理后台目录重新跑下 start.sh 脚本，心想这种问题重新运行下应该就没事了，想不到，竟然也折腾了一个小时，因此记录一下排查经历

过程

  1. 当重新运行了start.sh脚本后，系统能够正常运行，但是过一会就立马又是卡死状态，我打开浏览器登录进入系统，f12打开控制台看到获取菜单的请求一直在加载中，没有拿到响应，使用 tail 命令查看日志信息，请求日志和响应的日志信息都能够正常输出
  2. 心想这是什么玩意，响应也返回了为什么会卡死呢，猜测是不是网关出了问题，随即又去排查网关的日志，发现只输出了请求的日志，没有输出响应的日志，难道是网关的问题吗，随即remote debug网关服务(测试服务器在本地机房，为了方便调试开启了remote debug)，将断点打在了 NettyRoutingFilter#filter 方法中(NettyRoutingFilter是最终的发送请求的过滤器)，然后让客户端的请求发送过来，发现只有请求发送过去，迟迟接受不到系统服务的响应(没有配置responseTimeout配置，会一直等待服务的响应)，这肯定是系统服务出问题了
  3. 排查是不是系统的硬盘空间不足了导致的，使用 df -h 命令查看还有100多G的空间，当时就陷入了死胡同，过了一会儿突然想到可以用 jastck 命令打印下服务的线程信息，查看下请求线程的状态堆栈，随即就发现了问题根源所在

  1. 可以看到我们的请求线程已经被BLOCKED了，线程堆栈表明是rabbitmq消息发送阻塞住了，马上连接rabbitmq的服务器，打印日志发现磁盘不足导致消息根本无法写入，使用df -h 指令查看硬盘空间情况，发现只有几百k的空间了，因为这台服务器只运行了docker容器，猜测是docker的运行日志未清理导致空间占用的过大，然后编写了清理日志的脚本
title
#!/bin/sh 
  
echo "======== start clean docker containers logs ========"  
  
logs=$(find /var/lib/docker/containers/ -name *-json.log)  
  
for log in $logs  
        do  
                echo "clean logs : $log"  
                cat /dev/null > $log  
        done  

echo "======== end clean docker containers logs ========"

再刷新浏览器请求果然立马成功了，长吁一口气，不过由于docker会一直运行，产生的日志大小会源源不断的增长，为了治本设置了docker的日志大小上限，以免再出现这种情况

  1. 修改/etc/docker/daemon.json 文件，将下面的内容复制进去再 systemctl restart docker 即可
# 增加下面的内容
"log-driver":"json-file",
"log-opts": {"max-size":"1g", "max-file":"3"}
